# OpenCodeUI 项目自动同步与侧栏重构需求文档

日期：2026-03-01  
状态：待评审  
版本：v0.7

## 1. 目标与背景

当前 OpenCodeUI 的项目管理主要依赖本地 `savedDirectories`（用户手动添加目录），导致：

1. 首次使用需要逐个添加项目，启动成本高。
2. 项目列表依赖本地存储，不同设备间无法自然同步。
3. 侧栏信息架构偏“会话列表”，项目-会话层级不明显，不利于多项目场景。

本次需求目标：

1. 接入 OpenCode `/project` 接口，启动时自动初始化所有项目。
2. 以服务端项目列表为主数据源，实现跨端一致的项目视图。
3. 重构左侧“项目+会话”管理，交互与信息架构向 Codex 风格靠拢。
4. 保留现有单项目侧栏方式，同时新增多项目视图，通过顶部 Tab 在两种视图间切换。

## 2. 现状分析（OpenCodeUI）

### 2.1 当前实现要点

- `DirectoryContext` 使用本地存储键：`opencode-saved-directories`、`opencode-recent-projects`。
- 侧栏项目来自 `savedDirectories`，通过 “Add project...” 人工补充。
- 会话列表由 `SessionContext` 按当前 `directory` 拉取 `/session`，并显示扁平列表。
- 代码里已有 `/project` API（`getProjects/getCurrentProject`），但主流程未接入。

### 2.2 与目标差距

1. 项目来源不统一：本地目录列表 ≠ 服务端项目列表。
2. 缺少项目级树形会话视图（当前偏单目录筛选 + 扁平会话）。
3. 会话行快捷操作不足（目前主要是重命名/删除，缺少 Codex 风格上下文菜单）。

## 3. Codex 左侧栏图片分析（两张截图综合）

基于两张截图，提炼如下设计模式：

### 3.1 信息架构

1. 顶部有全局动作入口（如新线程等）。
2. 中段是“线程”工作区，核心结构为“项目 > 会话”的树形层级。
3. 项目与会话在同一滚动容器中连续展示，不分离为多个面板。

### 3.2 项目节点特征（第二张图补充）

1. 项目行在 hover 时，右侧显示操作区（`...` 菜单 + 新建会话图标）。
2. 新建会话图标用于“在该项目下直接创建会话”。
3. 项目菜单用于管理项目自身（如编辑名称、移除）。
4. 项目名称下可展示路径/目录等次级信息。
5. 项目展开后展示该项目下会话，形成明确归属关系。

### 3.3 会话节点特征

1. 会话行支持选中态高亮（胶囊底色）。
2. 右侧展示相对时间（如“1周”“2周”“3周”）。
3. 会话行 hover 时，左侧可出现 Pin 快捷按钮。
4. 当会话出现新的 notification 且用户不在该会话时，左侧显示蓝点提示。

### 3.4 交互模式

1. Web 场景不依赖右键；主要通过行尾 `...` 打开上下文菜单。
2. 桌面端可使用 hover 显示操作入口，保持界面简洁。
3. 移动端无 hover，需要可直接触达的菜单入口（点击 `...`）。
4. 线程区域右上角存在筛选图标，点击弹出筛选菜单。

### 3.5 可借鉴方向

1. 从“按时间平铺会话”转为“按项目组织会话”。
2. 将常见操作放在行内动作区，减少跳转与认知切换。
3. 在大量项目/会话场景下，依赖折叠与渐进加载控制信息密度。

### 3.6 已拍板决策（锁定）

1. 置顶会话在顶部“置顶区”展示，同时在原所属项目列表中保留。
2. 筛选条件不作用于置顶区，置顶区始终完整展示。
3. 蓝点在用户进入该会话后立即清除（等价于通知已消费）。
4. 默认项目采用“上次访问项目优先”。
5. 项目内“加载更多”步长固定为 3 条。
6. 侧栏提供两个 Tab：`多项目视图` 与 `单项目视图`，默认进入 `多项目视图`。
7. `单项目视图`仅作为兜底方案，保持现状，不做功能改造。

## 4. 需求范围

### 4.1 功能需求（FR）

### FR-00 侧栏双视图模式

1. 侧栏顶部提供两个 Tab：`多项目视图`、`单项目视图`。
2. 默认进入 `多项目视图`。
3. 切换视图不应打断当前会话，不应触发整页刷新。
4. `单项目视图`作为兜底入口保留原样，不新增定制化能力。

### FR-01 项目自动初始化（核心）

1. 应用启动、服务器切换后，调用 `GET /project` 拉取全部项目。
2. 将返回结果作为项目列表主数据源（以 `project.id/worktree` 唯一标识）。
3. 在`多项目视图`中不提供 `Global/All projects` 状态，始终在具体项目上下文中工作。
4. 若接口失败：
   - 显示非阻塞错误提示；
   - 回退到最近一次缓存项目列表（若有）；
   - 不阻断会话基础功能。

### FR-02 跨端同步语义

1. 项目来源以服务端 `/project` 为准，减少本地人工维护。
2. 新设备首次登录同一服务端时，自动看到已有项目。
3. 本地仅存储“UI状态”（如展开折叠、最近访问、临时置顶），不再作为项目真源。

### FR-03 多项目视图：项目-会话树

1. 左侧主工作区以项目为一级节点，会话为二级节点。
2. 支持项目展开/折叠（记忆状态）。
3. 项目和会话主列表上方提供“置顶区（Pinned）”。
4. 每个项目默认仅展示最近 3 条会话。
5. 当项目会话数 > 3 时，显示“加载更多/展开更多”入口，按批次或逐步展开。
6. 会话行显示：标题、相对更新时间；选中态高亮。
7. 交互目标：用户无需先“加项目”即可直接切换项目并管理会话。

### FR-04 会话上下文菜单（对齐 Codex 风格）

会话项通过 `...` 菜单提供操作，规则如下：

1. 桌面端：鼠标悬停会话行时显示右侧 `...`，左侧显示 Pin 快捷按钮。
2. 移动端：会话行右侧直接可点 `...`（无 hover 依赖），点击后使用底部弹层或弹出菜单。
3. 菜单动作首版仅包含：
   - 重命名线程（`PATCH /session/{sessionID}` title）
   - 归档线程（`PATCH /session/{sessionID}` time.archived）
   - 复制工作目录（session.directory）
4. Pin / Unpin 通过会话左侧快捷按钮触发（可选在菜单中保留同名入口），Pin 后会话进入顶部置顶区。
5. 明确不做（本期排除）：标记未读、复制会话 ID、复制 Deeplink、派生到本地、派生到新工作树。

### FR-05 项目节点菜单（对齐第二张图）

1. 项目行在 hover 时显示两个右侧动作：`...` 菜单 与“新建会话”图标。
2. 点击“新建会话”图标后，直接在当前项目目录下创建会话。
3. 项目 `...` 菜单动作：编辑名称、移除项目。
4. 移除项目需二次确认，避免误触。
5. 移动端沿用同一菜单语义，通过可点击按钮触发。

### FR-06 搜索与筛选

1. 支持会话搜索（保留现有搜索能力）。
2. 不提供“项目内/全局搜索”切换开关。
3. 在多项目视图中，搜索范围固定为“当前项目会话”。
4. 多项目视图线程区域右上角提供筛选图标，点击后展示筛选弹层，首版包含：
    - 整理：按项目 / 时间顺序列表
    - 排序条件：已创建 / 已更新
    - 显示：所有线程 / 相关

### FR-07 移动端排布与交互

1. 移动端侧栏采用抽屉式布局，树结构与桌面保持一致（项目 > 会话）。
2. 项目行/会话行触控命中区域不小于 44px 高度。
3. 会话右侧保留可直接点击的 `...`，避免依赖 hover。
4. 超长标题优先保证可点击区与时间信息不重叠，必要时省略路径次级信息。
5. “加载更多/展开更多”在移动端需一键可达，避免深层手势。
6. 项目行“新建会话”与 `...` 在移动端均需可点击，不依赖 hover。

### FR-08 兼容与降级

1. 在无项目或空项目时提供清晰空态文案。
2. 当 `/project` 与当前路由目录不一致时，优先保证会话可访问，再异步修正项目映射。
3. `/project` 拉取失败时，侧栏仍可基于缓存项目工作。

### FR-09 会话通知提示（蓝点）

1. 蓝点严格表示“该会话存在新的 notification”。
2. 仅当会话非当前选中会话且有未处理 notification 时，显示蓝点。
3. 用户进入该会话或对应 notification 被标记已读后，蓝点清除。

### FR-10 置顶区（Pinned）

1. 在整个“项目与会话区域”最上方增加置顶区。
2. 置顶区展示所有被 Pin 的会话（跨项目统一展示，不受项目折叠影响）。
3. 置顶区会话项可直接进入会话，并显示最小必要信息（标题、时间、所属项目标识）。
4. Unpin 后会话从置顶区移除，回到所属项目会话列表中。
5. 置顶区为空时默认折叠或隐藏占位，不影响主列表布局。

### FR-11 单项目视图（兼容模式）

1. 保留现有单项目侧栏信息架构与主要交互方式，作为兜底模式。
2. 单项目视图保持现状，不改动原有功能和交互。
3. 不要求为单项目视图适配多项目视图新增能力（如置顶区、项目树、筛选弹层）。
4. 当多项目视图异常时，用户可手动切换到单项目视图继续使用。

### 4.2 非功能需求（NFR）

1. 首屏项目加载不应显著拖慢现有启动速度（目标：项目请求失败不阻塞聊天入口）。
2. 切换项目到显示会话列表应保持流畅（避免全量重渲染）。
3. 交互反馈明确：加载中、空态、错误态、重试入口。
4. 视觉风格与现有 OpenCodeUI 主题系统兼容（不破坏浅色/深色主题一致性）。

## 5. 数据与状态设计（建议）

### 5.1 数据源分层

1. **服务端主数据**：`/project` 返回项目列表。
2. **本地缓存数据**：项目列表快照（仅用于离线/失败回退）。
3. **本地UI元数据**：
   - 项目展开状态
   - 项目最近访问时间
   - 会话 pin（若后端暂无字段，含置顶顺序）
   - 会话 notification 蓝点消费状态（按会话ID记录最近已处理时间戳）

### 5.2 现有上下文改造建议

1. `DirectoryContext`：从“本地目录中心”转为“当前项目选择 + UI偏好”。
2. `SessionContext`：按选中项目目录加载会话。
3. `SidePanel`：升级为双视图容器，保留旧单项目视图并新增多项目视图。

## 6. 页面与交互重构要点

1. 侧栏顶部提供 `多项目视图` / `单项目视图` 两个 Tab。
2. 头部保留“新会话”主动作，减少路径层级。
3. 多项目视图中，“项目列表”与“会话列表”合并为一个树。
4. 项目节点点击行为：
   - 单击项目名：切换项目并展开；
   - 展开箭头：仅控制展开折叠。
5. 会话节点点击行为：切换会话并同步路由。
6. 置顶区位于项目树上方，优先展示被 Pin 会话。
7. 桌面端会话行 hover：左 Pin、右 `...`；项目行 hover：右 `...` + 新建会话图标。
8. 多项目视图线程区右上角放置筛选图标，点击弹出筛选菜单。

## 7. 验收标准（AC）

### AC-00 双视图切换

1. 侧栏顶部存在 `多项目视图` / `单项目视图` 两个 Tab。
2. 默认进入 `多项目视图`。
3. 切换视图不打断当前会话上下文。

### AC-01 项目自动同步

1. 新安装/新设备首次打开，项目列表可自动出现（无需手动 Add project）。
2. 切换服务器后，项目列表跟随服务器隔离刷新。
3. 在多项目视图中无 `Global/All projects` 入口，用户始终在具体项目中工作。
4. `/project` 失败时，界面可降级并给出重试。

### AC-02 多项目视图信息架构

1. 侧栏可清晰看到“项目 > 会话”层级。
2. 置顶区位于项目树上方，显示被 Pin 的会话。
3. 可展开/折叠项目，并记忆状态。
4. 每个项目默认显示 3 条会话，存在更多时可“加载更多/展开更多”。
5. 会话可显示相对时间、选中态。

### AC-03 会话操作

1. 会话菜单仅包含：重命名、归档、复制工作目录。
2. 桌面端通过 hover 显示 `...`；点击 `...` 可打开菜单（无右键依赖）。
3. 移动端可直接点击 `...` 打开菜单（底部弹层或弹出菜单）。
4. 会话行 hover 左侧出现 Pin 按钮，可快速 Pin/Unpin。

### AC-04 项目操作

1. 项目行 hover 显示 `...` 与“新建会话”图标。
2. 点击“新建会话”可在该项目下创建会话。
3. 项目菜单包含：编辑名称、移除。
4. 项目移除具备二次确认。

### AC-05 可用性

1. 桌面端和移动端均可完成项目切换、会话切换、菜单操作。
2. 大量项目/会话下滚动与切换无明显卡顿。
3. 移动端抽屉中树结构可顺畅展开与收起。
4. 多项目视图线程区右上角筛选图标可打开筛选弹层并生效。

### AC-06 会话通知提示

1. 非当前会话出现新的 notification 时，左侧显示蓝点。
2. 进入会话或 notification 已读后蓝点消失。

### AC-07 置顶区

1. Pin 任意会话后，会话出现在顶部置顶区。
2. 置顶区可跨项目展示 Pin 会话，并可直接点击进入。
3. Unpin 后会话从置顶区移除。
4. 置顶会话在原所属项目列表中仍保留。

### AC-08 单项目视图兼容

1. 单项目视图保留原有主要交互路径，老用户可无学习成本继续使用。
2. 单项目视图维持现状，不承载多项目视图新增能力。
3. 多项目视图出现问题时，用户可切到单项目视图完成核心会话操作。

## 8. 依赖与风险

1. 当前代码虽定义 `project.updated` 事件类型，但事件管道尚未接入，实时同步可能需补齐 SSE 映射。
2. Pin 若无后端字段，首版只能本地持久化；跨端一致需后续接口支持。
3. 会话蓝点依赖 notification 事件与已读状态同步，需要处理去重、丢事件与重连补偿。
4. 项目会话默认仅展示 3 条后，需要设计增量加载策略，避免频繁全量请求。
5. 当前 OpenAPI 仅提供 `/experimental/worktree`（sandbox worktree）能力，未见“永久工作树”独立字段或接口，本期不纳入。
6. 单项目视图保持原样时，需防止多项目改造对旧视图造成回归影响。

## 9. 分期建议

### Phase 1（高优先级）

1. 接入 `/project` 自动初始化与失败降级。
2. 侧栏顶部上线 `多项目视图` / `单项目视图` Tab，并保留旧单项目视图。
3. 上线多项目视图项目-会话树（展开、默认 3 条 + 加载更多）。
4. 上线会话行 hover 交互（左 Pin、右 `...`）与会话菜单（重命名、归档、复制工作目录）。
5. 上线项目行 hover 交互（右 `...` + 新建会话图标）与项目菜单（编辑名称、移除）。
6. 上线多项目视图右上角筛选图标与筛选弹层。
7. 移动端完成无 hover 菜单与操作按钮交互（点击 `...` / 新建会话）。
8. 上线会话 notification 蓝点提示机制。
9. 上线置顶区（Pinned）及 Pin/Unpin 联动。

### Phase 2（中优先级）

1. Pin 本地持久化与多端同步策略评估。
2. 蓝点提示规则与状态来源优化（减少误报/漏报）。
3. 细化排序、筛选、项目管理辅助能力。

### Phase 3（增强）

1. 接入项目实时更新事件（如 `project.updated`）。
2. 评估将 Pin 升级为服务端能力，实现真正跨端一致。

## 10. 实施方案（迭代版）

### 10.1 方案目标

1. 以最小风险上线`多项目视图`，并默认启用。
2. 保留`单项目视图`作为兜底，不改造旧逻辑。
3. 优先保证通知蓝点、Pin 置顶、会话切换三条关键链路稳定。

### 10.2 设计原则

1. **增量改造**：新增“多项目视图”而非替换旧视图。
2. **状态单源**：通知状态、会话状态、Pin 状态保持唯一来源，避免双视图状态漂移。
3. **先可用后完美**：先打通主路径，再细化排序、筛选和体验。
4. **可回退**：任何阶段都能快速切回`单项目视图`继续使用。

### 10.3 总体策略

1. 先搭建双视图壳层与默认策略（默认多项目，单项目兜底）。
2. 再落多项目基础信息架构（项目树、默认 3 条、加载更多）。
3. 再接入行为链路（项目新建会话、会话菜单、通知蓝点、Pin 置顶）。
4. 最后收口筛选、移动端交互与稳定性回归。

### 10.4 稳定性与回退策略

1. 通过“视图 Tab”形成用户侧可见回退路径。
2. 若多项目视图出现问题，用户可立即切换到单项目视图。
3. 多项目视图发布后，优先监控：会话打开成功率、通知蓝点正确率、会话创建成功率。
4. 发生异常时，以“保核心链路”为优先：会话可打开 > 通知可见 > 视觉一致性。

### 10.5 关键联动约束

1. 蓝点严格绑定 notification，不与“未读消息”混用。
2. Pin 后同时出现在置顶区与原项目列表。
3. 进入会话后，蓝点即时清除。
4. 筛选仅影响主列表，不影响置顶区。
