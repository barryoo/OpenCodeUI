# OpenCodeUI 服务端化与事项管理 PRD

日期：2026-04-11  
状态：待评审  
版本：v1.0

## 1. 背景

当前 OpenCodeUI 是纯前端应用，核心数据依赖浏览器本地存储，存在以下问题：

1. 同一用户在不同设备访问同一域名时，无法同步服务器配置。
2. 服务器配置、事项以及会话绑定能力缺少后端持久化基础。
3. 当前界面以会话为中心，缺少“事项”这一上层管理对象，无法支持需求、Bug、研究、Code Review 等长期事项推进。

本次需求目标是在现有 OpenCodeUI 基础上增加一层轻量服务端能力，并引入事项管理，形成“用户 -> 服务器 -> 项目 -> 事项/会话”的统一模型。其中项目和会话详情仍由 OpenCode 负责，新增服务端仅管理服务器配置、事项，以及会话摘要与事项绑定关系。

## 2. 目标

### 2.1 产品目标

1. 支持多用户登录，用户数据按账号隔离。
2. 支持服务器配置保存在服务端，并在多端同步。
3. 支持服务器维度的项目视图组织。
4. 支持项目下的事项管理，以及会话摘要与事项绑定管理。
5. 支持事项与会话一对多关联，满足一个大事项分多次会话推进的使用方式。
6. 在 UI 上明确区分三类工作区域：左侧导航、中间事项详情、右侧会话详情。

### 2.2 V1 范围

1. GitHub 登录。
2. 后端技术栈采用 Node/Bun。
3. 数据库存储采用 SQLite。
4. 事项描述支持 `@` 引用当前项目文件。
5. 不支持导入旧浏览器本地数据。

## 3. 非目标

1. 不做团队共享、多人协作权限。
2. 不做事项/会话操作历史列表。
3. 不做按事项筛选会话、未绑定会话筛选。
4. 不做旧本地数据自动迁移。
5. 不在左侧把事项和会话做树状展开展示。

## 4. 角色与数据归属

### 4.1 用户

1. 系统支持多用户。
2. 用户通过 GitHub 登录进入系统。
3. 各用户之间数据完全隔离。

### 4.2 数据归属

1. 服务器归属于用户。
2. 项目逻辑上归属于服务器，但项目数据真源来自 OpenCode。
3. 事项归属于项目。
4. 会话详情归属于 OpenCode，服务端仅保存与事项功能相关的会话摘要。
5. 会话可选绑定事项，允许无事项存在。

## 5. 核心对象定义

### 5.1 服务器

服务器表示一个可连接的 OpenCode/OpenClaw/OpenCore 服务端配置。

基础字段建议包括：

1. 名称
2. 地址
3. 协议/端口
4. 鉴权信息
5. 是否默认
6. 创建时间/更新时间

### 5.2 项目

项目是服务器下的一级工作容器，但项目列表与项目基础信息仍以 OpenCode 为准。

基础字段建议包括：

1. 名称
2. 目录/工作区标识
3. 所属服务器
4. 创建时间/更新时间

### 5.3 事项

事项用于承载一个长期目标或待处理对象。

事项类型：

1. 需求
2. Bug
3. 研究
4. Code Review

基础字段建议包括：

1. 标题
2. 类型
3. 状态
4. 详细描述
5. 所属项目
6. 最近活动时间
7. 创建时间/更新时间

### 5.4 会话摘要

会话本体仍由 OpenCode 管理。新服务端仅保存与事项功能相关的会话摘要。

基础字段建议包括：

1. OpenCode 会话 ID
2. 标题摘要
3. 状态摘要
4. 所属项目标识
5. 可选事项绑定
6. 最近活动时间
7. 创建时间/更新时间

## 6. 数据关系

1. 一个用户可拥有多个服务器。
2. 一个服务器下可有多个项目。
3. 一个项目下可有多个事项。
4. 一个项目下可有多个会话。
5. 一个事项可关联多个会话。
6. 一个会话最多绑定一个事项。
7. 一个会话可以没有事项。
8. 一个会话允许从事项 A 改绑到事项 B。

## 7. 功能需求

### FR-01 GitHub 登录

1. 未登录用户访问时进入登录态页面。
2. 用户通过 GitHub OAuth 登录。
3. 登录成功后创建或更新用户记录。
4. 前端可获取当前用户信息与登录状态。

### FR-02 服务器配置服务端持久化

1. 用户可新增、编辑、删除服务器配置。
2. 用户可设置一个默认服务器。
3. 用户在任意设备登录后，可看到自己全部服务器配置。
4. 前端进入应用后优先拉取服务端服务器配置，而非仅使用本地存储。
5. 默认服务器存在时，前端自动以该服务器作为当前目标。

### FR-03 服务器下项目视图

1. 系统支持按当前服务器展示项目列表。
2. 项目列表由 OpenCode 提供，服务端不作为项目真源。
3. 项目列表是左侧导航的第一层组织单位。
4. 切换服务器后，项目上下文一起切换。

### FR-04 左侧导航展示规则

1. 左侧显示项目列表。
2. 进入某个项目后，左侧展示该项目下的混排列表。
3. 混排列表中的事项和会话不做明显分区。
4. 如果会话没有绑定事项，则直接显示该会话条目。
5. 如果会话已经绑定事项，则左侧只显示该事项条目，不重复显示其子会话。
6. 列表统一按展示顺序规则排序，与类型无关。
7. 每一行右侧显示该事项或会话的状态。

### FR-05 三栏/两栏布局

1. 默认布局支持三栏：左栏、中栏、右栏。
2. 左栏为项目与混排列表。
3. 中栏为事项详情与事项下会话列表。
4. 右栏为会话详情，且为主工作区，占据最大宽度。
5. 当用户点击左栏中的事项时，打开三栏模式。
6. 当用户点击左栏中的未绑定事项会话时，直接打开会话详情，界面切换为左右两栏，中栏隐藏。

### FR-06 事项详情展示与编辑

1. 点击事项后，中栏上半区显示事项详情。
2. 事项详情至少包含标题、类型、状态、更新时间、描述。
3. 事项详情支持就地编辑，不使用单独弹窗。
4. 用户可在事项进行过程中持续编辑描述。
5. 事项描述更新后保存到服务端。

### FR-07 事项描述中的 @ 文档引用

1. 用户在事项描述输入框中输入 `@` 时，触发候选提示。
2. 候选来源为当前项目下的文件。
3. 用户选择候选后，将对应文件引用插入描述内容。
4. 系统保存事项描述正文以及引用结果。

### FR-08 事项下会话列表

1. 点击事项后，中栏下半区显示该事项的会话列表。
2. 会话列表为竖向列表。
3. 每条会话显示标题、状态、最近活动时间。
4. 在事项上下文中可直接新建 OpenCode 会话，新建后默认自动绑定当前事项。
5. 点击某条会话后，右栏切换为该会话详情。
6. 中栏顶部不提供“返回事项”按钮。

### FR-09 会话摘要管理与绑定

1. 用户可直接在项目下创建 OpenCode 会话，不要求必须先创建事项。
2. 服务端仅记录会话摘要与事项绑定关系，不保存会话正文。
3. 用户可把无事项会话后续绑定到某事项。
4. 用户可把已绑定事项 A 的会话改绑到事项 B。
5. 一个事项下允许存在多个会话。

### FR-10 状态规则

事项状态与会话状态均支持：

1. 未开始
2. 进行中
3. 已完成
4. 已放弃

状态规则如下：

1. 会话摘要状态根据 OpenCode 会话活动自动进入进行中。
2. 若某事项下任一会话处于进行中，则事项自动进入进行中。
3. 若现有会话在首次纳入摘要管理时，其最后更新时间已超过 14 天，则系统自动标记为已完成。
4. 运行期间，如果会话持续超过 14 天没有任何变化，摘要状态也应自动变为已完成。
5. 已完成与已放弃允许恢复到进行中。

### FR-11 历史兼容

1. 现有历史会话允许没有事项绑定。
2. 没有事项绑定的会话仍可正常展示与打开。
3. 不要求把旧本地数据自动导入后端。
4. V1 以后服务器配置、事项与会话摘要数据以服务端为准；项目与会话详情仍以 OpenCode 为准。

## 8. 页面交互说明

### 8.1 左栏点击事项

1. 左栏保持项目与混排列表。
2. 中栏打开当前事项。
3. 中栏上方展示事项详情。
4. 中栏下方展示该事项的会话列表。
5. 右栏展示当前选中的会话详情；若尚未选择事项下会话，则右栏可为空态。

### 8.2 左栏点击未绑定事项会话

1. 中栏隐藏。
2. 右栏直接展示该会话详情。
3. 整体界面变为两栏。

### 8.3 在事项中编辑描述

1. 用户直接在中栏上方的事项详情区编辑。
2. 编辑时支持普通文本输入与 `@` 文件引用。
3. 保存成功后更新事项更新时间与活动时间。

## 9. 排序与活动时间

1. 左栏混排列表由统一排序规则控制。
2. 建议默认规则为按最近活动时间倒序。
3. 事项最近活动时间可由事项自身更新时间或其下最近会话活动时间共同决定。
4. 会话最近活动时间取会话自身最后更新时间。

## 10. 验收标准

### AC-01 多端同步

1. 用户在设备 A 新增服务器配置后，设备 B 登录可见同一配置。
2. 默认服务器在多端一致。

### AC-02 结构与关系

1. 用户可在服务器下看到多个项目。
2. 项目中可创建事项与会话。
3. 一个事项可关联多个会话。
4. 会话可无事项，也可后续绑定事项。

### AC-03 UI 布局

1. 点击事项后进入三栏布局。
2. 中栏展示事项详情与事项会话列表。
3. 点击未绑定事项会话后进入两栏布局。
4. 右栏始终是主要会话工作区。

### AC-04 状态联动

1. 会话开始后自动进入进行中。
2. 事项下存在进行中会话时，事项自动进入进行中。
3. 超过 14 天未更新的会话自动完成。
4. 已完成或已放弃状态可恢复到进行中。

### AC-05 文档引用

1. 事项描述输入 `@` 可弹出当前项目文件候选。
2. 选择候选后可写入事项描述。

## 11. 风险与后续

1. GitHub 登录引入后，需要明确 Web 部署时的回调地址与会话管理策略。
2. `@` 文件引用如果直接扫描全项目，需关注性能与搜索体验。
3. V1 不做旧数据导入，升级后旧浏览器本地数据不会自动出现在后端。

后续可扩展方向：

1. 多用户共享与协作权限。
2. 事项/会话操作历史。
3. 会话筛选、固定、搜索增强。
4. 文档引用的富文本与预览能力。

## 12. 当前未完成功能清单（开发跟踪）

以下为截至当前版本仍**未完成**或**未达到可验收状态**的功能：

### 12.1 登录与用户体系

1. GitHub OAuth 虽已接入后端骨架，但**尚未完成完整验收**（跳转、回调、生产环境配置、异常处理、跨域部署验证未闭环）。
2. 当前前端仍包含 demo 登录兜底逻辑，**正式多用户登录体验未完成**。
3. 用户登录后的全局状态管理、登录态失效处理、登录后初始化流程尚未完整收口。

### 12.2 Server 配置多端同步

1. Settings > Servers 已接入薄后端作为真源，但**尚未完成完整手工验收**。
2. 多设备登录后 server 配置同步的真实场景尚未验证。
3. server 切换后对事项/会话摘要数据刷新的联动稳定性尚未完全验证。

### 12.3 主界面事项三栏改造

1. 现有主界面已开始接入事项中栏，但**整体交互尚未完全收口**。
2. 左侧“事项 + 未绑定会话”混排虽已接入，但排序、状态展示、选中态、空态、失败态仍需进一步打磨与验证。
3. 点击事项进入三栏、点击未绑定会话进入两栏的完整交互链路尚未完成最终验收。
4. 中栏事项详情与右侧会话详情联动仍需继续稳定化处理。

### 12.4 事项管理闭环

1. 事项创建、编辑、状态更新已部分接入，但**完整可用性尚未确认**。
2. 一个事项下多个会话的管理体验仍需继续打磨。
3. 会话从事项 A 改绑到事项 B 的完整 UI 操作链尚未完成。

### 12.5 事项描述中的 @ 文档引用

1. 已接入项目文件搜索能力，但**引用关系持久化（item_document_refs）尚未完全打通到前端保存链路**。
2. `@` 候选的展示、插入、编辑后再保存的完整体验仍需继续验证。

### 12.6 状态机与自动规则

1. 会话摘要超 14 天自动完成逻辑已写入后端，但**真实运行期调度/定时任务尚未完善**。
2. 事项状态自动联动为进行中已部分实现，但边界场景（解绑、恢复、全部完成后状态）尚未完全验证。

### 12.7 联调与验收

1. 主界面事项能力、Settings > Servers、登录态三者之间的整体联调尚未完整完成。
2. 尚未完成一轮覆盖主要用户路径的手工验收：
   - 登录
   - 添加/编辑/删除 server
   - 选择默认 server
   - 项目内创建事项
   - 事项关联会话
   - 中栏/右栏联动
3. 尚未形成“已完成可提交”的状态。
