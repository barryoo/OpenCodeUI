# 会话运行状态调研记录

## 现象

- 会话运行时，侧栏会显示三个跳动小点
- 刷新页面后，这个小点经常消失
- 有时点进会话后，过一会儿状态又会回来

## 当前实现结论

- 侧栏小点来自 `activeSessionStore`
- `activeSessionStore` 的状态来源是：`/session/status`、`/permission`、`/question` 和 SSE 事件
- OpenCode Server 的 `/session/status` 不是稳定的“完整会话状态真相源”

## 已确认的关键点

### 1. `/session/status` 可能返回空对象

实测中，这个接口在会话仍然运行时也可能返回空对象。

结合 OpenCode 上游信息，可以确认：

- `/session/status` 只表示当前活跃的内存状态
- 空对象不一定等于“所有会话都已经结束”
- 某些异步场景下，这个状态接口本身就可能不稳定

相关上游线索：

- `anomalyco/opencode#12860`
- `anomalyco/opencode#12870`
- `anomalyco/opencode#12922`
- `anomalyco/opencode#13113`

### 2. 当前 UI 会把这个接口结果放大成明显问题

如果前端在刷新后把 `/session/status` 返回值直接当成唯一真相，那么：

- 一次空返回就会把侧栏 running 状态清空
- 后续只有等新的 SSE 或其他补充信号到达，小点才会重新出现

这正好符合“刷新后消失，过一会儿又出现”的现象。

### 3. 问题未必只在前端

这不是单纯的前端本地状态丢失。

更准确地说，是：

- 前端确实依赖内存态展示 running
- 但这个内存态本来应该由服务端状态和 SSE 恢复
- 而服务端状态接口本身并不总是稳定可靠

## 本次尝试结果

本次曾尝试加入“刷新桥接恢复”逻辑，但用户验证后反馈没有实际效果，因此没有保留这套实现代码，只保留调研记录。

## 后续建议

如果后面继续修，建议先不要直接堆前端缓存逻辑，而是优先做下面两步：

1. 在真实运行环境里抓刷新前后 `/session/status` 与 SSE 事件时序，确认到底是哪一层先缺失
2. 确认 OpenCode 当前版本是否已经包含 `prompt_async` / session status 相关上游修复

只有先把“服务端状态是否稳定可依赖”确认清楚，再决定前端到底要不要做桥接恢复，才比较稳妥。
